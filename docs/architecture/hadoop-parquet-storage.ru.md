# Хранение Parquet-файлов и локальность

Этот документ описывает, как проект работает с файлами через Hadoop FileSystem, как используется информация о локальности HDFS-блоков и какие ограничения есть у текущей модели распределения.

## Модель хранения

Данные не хранятся внутри Arrow Flight сервера. Flight-ноды являются compute layer: они получают назначенные Parquet-файлы и читают их через Hadoop FileSystem.

Корень данных задается параметром `dataDirectory`. Таблицы ожидаются в структуре `schema/table`. Для каждой таблицы сервер рекурсивно ищет Parquet-файлы.

Основная логика работы с файлами находится в `ParquetManager`.

## Hadoop FileSystem

Hadoop FileSystem используется как общий слой доступа к данным. Через него сервер может читать HDFS, локальную файловую систему или другой Hadoop-совместимый backend.

Flight-нода не обязана иметь Parquet-файл на локальном диске. Если файл лежит в распределенной файловой системе, нода открывает его через Hadoop FileSystem.

При старте сервер создает `FileSystem`, связанный с `dataDirectory`. Все операции с файлами проходят через этот объект: listing, чтение metadata, получение абсолютных путей и открытие потоков.

## HDFS и локальность блоков

В HDFS файл разбивается на блоки. Блоки могут лежать на разных DataNode и обычно реплицируются. Hadoop может вернуть список host-ов, где находятся блоки конкретного файла.

Проект использует эту информацию для data locality. Если Flight server запущен на host, где лежат блоки файла, чтение этого файла с такой ноды предпочтительнее.

`ParquetManager.fileLocality` извлекает host-ы из block locations, возвращенных Hadoop.

## Поиск файлов для запроса

Когда серверу нужно понять, какие файлы относятся к запросу, он вызывает `ParquetManager.locationsForQuery`.

Метод:

1. Разбирает SQL и определяет schema/table.
2. Строит путь таблицы под `dataDirectory`.
3. Рекурсивно ищет Parquet-файлы.
4. Собирает hosts, на которых лежат блоки каждого файла.
5. Возвращает mapping относительного пути файла к набору hosts.

Эта информация еще не означает, что файл назначен конкретной Flight-ноде. Она только показывает, какие ноды потенциально ближе к данным.

## Parquet как формат хранения

Parquet - колоночный формат. Это важно для аналитических запросов: сервер может читать только нужные колонки, а не все строки целиком.

Parquet-файл также содержит footer с metadata: schema, row counts, row group statistics и другие данные. Проект использует footer metadata для fast path оптимизаций.

Текущее распределение выполняется на уровне целых файлов. Row group-ы пока не используются как единица распределения между Flight-нодами.

## Projection

Projection означает выбор только нужных колонок. Для обычного `SELECT` parser извлекает список колонок и передает его в Arrow Dataset scanner.

Если запросу нужна только часть колонок, Acero читает только эти колонки. Это снижает I/O и объем данных, которые нужно передать в Arrow batches.

Если projection нет, scanner читает все колонки таблицы.

## Filter pushdown

Если запрос содержит `WHERE`, проект направляет его в DuckDB, а не в Acero. DuckDB читает назначенные Parquet-файлы через `read_parquet([...])` и применяет SQL predicate в своем Parquet reader.

Для HDFS paths DuckDB должен загрузить DuckDB HDFS extension. Путь к extension и HDFS options задаются через `arrowflight.properties`, JVM system properties или environment variables.

Такая маршрутизация выбрана намеренно: Acero используется для filter-less full scan и projection, а DuckDB - для filtered scans, потому что его Parquet reader умеет predicate pushdown и row-group pruning.

## Footer fast path

Некоторые агрегации можно выполнить без чтения data pages.

Для `COUNT(*)` проект использует row counts из Parquet footer. Для `COUNT(col)`, `MIN(col)` и `MAX(col)` проект пытается использовать row group statistics.

Если нужная статистика доступна и запрос не содержит фильтра или группировки, сервер читает только metadata. Если статистики недостаточно, выполнение fallback-ится в DuckDB.

Текущий fallback для обычных агрегаций - DuckDB. Acero не используется для grouped или filtered aggregation execution.

## Распределение файлов между Flight-нодами

Распределение выполняется в `HadoopFlightSqlService.determineEndpoints`.

Для каждого Parquet-файла сервер выбирает Flight-ноду, которая будет его читать.

Логика выбора:

1. Если есть Flight-ноды на hosts, где лежат блоки файла, рассматриваются только они.
2. Среди кандидатов выбирается нода с меньшей накопленной нагрузкой по размеру назначенных файлов.

После выбора файл добавляется в группу этой ноды, а нагрузка ноды увеличивается на размер файла. Затем для каждой ноды создается endpoint и ticket.

## Ограничения распределения

Текущая greedy-стратегия балансирует объем данных по размеру файлов, но работает на уровне целых файлов. Очень большой одиночный файл все еще будет назначен одной ноде.
