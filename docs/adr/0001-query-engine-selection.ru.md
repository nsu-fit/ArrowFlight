# ADR 0001: Выбор движка запросов в зависимости от типа операции

Status: Accepted
Date: 2026-07-06

## Решение

- Timestamp: 2026-07-06T12:00:00+07:00
- User: s.samokhin
- Вопрос: Какие движки использовать для чтения Parquet в Flight SQL сервере,
  учитывая что Acero (Arrow Dataset JNI) и DuckDB имеют разные профили
  производительности?
- Решение: Направлять каждый запрос в DuckDB или Acero в зависимости от
  распарсенной SQL-структуры. Full scan и проекции колонок → Acero. Запросы
  с фильтрами → DuckDB.
- Обоснование: Acero быстрее на полном сканировании и проекциях благодаря
  zero-copy чтению Parquet в Arrow. DuckDB быстрее на запросах с фильтрами
  благодаря push-down предикатов в Parquet reader (row-group pruning). Ни
  один движок сам по себе не оптимален для всех типов запросов.

## Контекст

Сервер поддерживает Flight SQL запросы над Parquet файлами. Доступны два
бэкенда для чтения:

- **Acero** (Apache Arrow Dataset JNI): читает Parquet напрямую в Arrow
  RecordBatch через нативный C++ код. Данные пересекают JNI границу через
  zero-copy Arrow C Data Interface. SQL движок не задействован.
- **DuckDB** (через JDBC): парсит SQL через jOOQ, исполняет через C++
  движок DuckDB, конвертирует результат в Arrow. Полная SQL семантика:
  фильтры, проекции, агрегации.

Изначально Acero был удалён из кодовой базы (ветка `duckdb-only-no-acero`),
потому что добавлял сложности (прогрев Substrait, нативные JNI библиотеки,
отдельная логика push-down фильтров) и падал с `NullPointerException` в
`ScanOptions.getColumns()`. Предположение было: "DuckDB умеет всё то же
самое, что и Acero — уберём Acero."

## Переоценка

Производительность была сравнена в бенчмарке (`ArrowFlightPerfTest`) на
1000-строчном Parquet файле (один snappy-фрагмент, 13 колонок,
Spark-сгенерированный):

| Сценарий | Acero (ms) | DuckDB (ms) | Победитель |
|---|---|---|---|
| Full scan (`SELECT *`) | 38 | 66 | Acero (1.7x) |
| Проекция колонок (3 шт.) | 26 | 76 | Acero (2.9x) |
| Фильтр (`WHERE tinyint_col = 0`) | 234 | 67 | DuckDB (3.5x) |
| Фильтр + проекция | 231 | 65 | DuckDB (3.6x) |
| Составной фильтр | 217 | 63 | DuckDB (3.4x) |

### Почему Acero выигрывает на full scan и проекциях

Acero читает Parquet-страницы напрямую в Arrow RecordBatch через нативный
C++ Parquet reader. Нет SQL-парсинга, нет построения плана запроса, нет
построчной конвертации. Для проекций Acero читает только запрошенные
колонки. Результат уже в формате Arrow — zero copy. DuckDB должен
распарсить SQL, построить план, прочитать Parquet в свой внутренний
формат, потом сконвертировать в Arrow. Каждый шаг добавляет накладные
расходы.

### Почему DuckDB выигрывает на фильтрах

DuckDB пушит фильтры в Parquet reader: читает метаданные каждой row-group
(минимум/максимум колонок) и пропускает группы, где значение не может
попасть под условие — row-group pruning. Acero в этой версии не пушит
фильтры — читает все row-group, материализует все строки в Arrow-батчи,
потом отбрасывает неотфильтрованные строки в памяти.

### Оба движка равны на Flight сервере

При чтении через Arrow Flight SQL (gRPC) оба движка показывают ~420-470ms.
Узкое место — gRPC round-trip и сериализация/десериализация Arrow, а не
чтение Parquet. Выбор движка влияет только на локальный (прямой) путь
чтения.

## Решение

- Классифицировать каждый SQL запрос в момент парсинга (до выполнения):
  1. **Full scan** (`SELECT *`, без фильтра) → Acero
  2. **Проекция колонок** (`SELECT a, b, c`, без фильтра) → Acero
  3. **Сканирование с фильтром** (`WHERE ...`) → DuckDB
  4. **Агрегация** (`GROUP BY`, `COUNT`, `SUM`, etc.) → DuckDB (Acero не
     поддерживает агрегации)

- Классификация дешёвая: `ParquetQueryParser.parseQuery()` уже извлекает
  `columns` (пусто = `*`), `filter` (пусто/не пусто) и `hasAggregation`
  до того, как начнётся I/O.

- `ParquetManager` предоставляет два метода:
  - `scanWithAcero(paths, columns)` — zero-copy Arrow сканирование
  - `scanWithDuckDB(paths, columns, filter)` — DuckDB SQL с push-down
    фильтров

- Прогрев: оба движка прогреваются при старте. Acero сканирует один Parquet
  файл для инициализации JNI. DuckDB прогревает 8 thread-local соединений
  (пул). Прогрев Substrait-конвертера пропущен (не нужен, когда Acero
  используется только для запросов без фильтров).

## Рассмотренные альтернативы

### Вариант A: Только Acero (всегда читать через Acero)

Использовать Acero JNI для всех запросов, включая фильтрованные. Это
упрощает кодовую базу до одного движка. Однако Acero не пушит фильтры в
Parquet reader — читает все row-group и фильтрует в памяти. Запросы с
фильтрами в 3-4x медленнее DuckDB. Агрегации не поддерживаются, требуется
отдельный fallback или вычисления на клиенте.

- **Плюсы**: Один движок, zero-copy Arrow вывод, максимальная скорость
  на full scan и проекциях.
- **Минусы**: Медленные фильтры, нет агрегаций, требуется прогрев
  Substrait-конвертера.

### Вариант B: Только DuckDB (всегда читать через DuckDB)

Использовать DuckDB JDBC для всех запросов. DuckDB поддерживает полную
SQL семантику: фильтры, проекции, агрегации. Это было состояние после
удаления Acero в ветке `duckdb-only-no-acero`.

- **Плюсы**: Один движок, полная SQL поддержка, push-down фильтров, не
  требует нативных JNI библиотек.
- **Минусы**: Full scan в 1.7x медленнее Acero, проекции в 2.9x медленнее.
  Каждый запрос проходит через SQL-парсинг, построение плана и Arrow
  конвертацию.

### Вариант C: Гибрид (выбран)

Направлять каждый запрос в лучший движок на основе распарсенной SQL.
Full scan и проекции колонок → Acero. Фильтрованные запросы и агрегации
→ DuckDB.

- **Плюсы**: Лучшая производительность для каждого типа запроса, ни один
  движок не перегружен работой, которую другой делает быстрее.
- **Минусы**: Два движка для поддержки, эвристическая классификация может
  ошибиться на краевых случаях (например, `WHERE 1=1`).

## Последствия

- **Положительное**: Каждый тип запроса использует самый быстрый доступный
  движок. Full scan по скорости близок к прямому чтению Parquet, запросы
  с фильтрами — к оптимизированному SQL движку.
- **Положительное**: Ни один движок не является мёртвым грузом. Acero
  сохранён для своей ниши. DuckDB не перегружен full scan работой, где он
  в 1.7x медленнее.
- **Отрицательное**: Два движка нужно поддерживать, тестировать и
  синхронизировать. Баги, специфичные для движка, проявляются только на
  определённых типах запросов.
- **Отрицательное**: Классификация на этапе парсинга — эвристика. Запрос
  `SELECT * FROM t WHERE 1=1` будет направлен в DuckDB (фильтр не пуст),
  хотя мог бы использовать Acero. Это приемлемый false positive —
  константные фильтры редки.
- **Будущее**: Если Acero получит нативный push-down фильтров (Arrow
  Dataset поддерживает `SubstraitFilter` в новых версиях), логика
  упростится: "всегда предпочитать Acero, падать на DuckDB для агрегаций".
