# ADR 0003: Выбор бенчмарк-набора — TPC-H / TPC-DS через BenchBase

Status: Accepted
Date: 2026-07-10

## Решение

- Timestamp: 2026-07-10T14:00:00+07:00
- User: s.samokhin
- Вопрос: Какой набор бенчмарков использовать для измерения производительности
  запросов, валидации корректности и отслеживания регрессий на разных движках
  Parquet?
- Решение: Использовать TPC-H и TPC-DS, запускаемые через фреймворк BenchBase.
  TPC-H покрывает decision-support запросы с широким доступом к колонкам;
  TPC-DS добавляет скошенные данные, вложенные подзапросы и оконные функции,
  характерные для реальных аналитических нагрузок.
- Обоснование: TPC-H и TPC-DS — индустриальные стандарты для decision-support
  бенчмарков. BenchBase берёт на себя загрузку запросов, привязку параметров,
  сбор метрик и валидацию результатов. Комбинация покрывает как простые
  scan-heavy запросы (TPC-H), так и сложные многостадийные запросы (TPC-DS),
  что соответствует целевой аналитической нагрузке проекта.

## Контекст

Проект выполняет Flight SQL запросы над Parquet файлами через три пути:
DuckDB (фильтрованный scan, агрегации), Acero (полный scan, проекции колонок)
и Java Parquet-footer fast path (агрегаты только по metadata). Необходимо
верифицировать производительность и корректность на всех трёх путях.

Набор бенчмарков нужен для:

- отслеживания задержки запросов и пропускной способности между релизами;
- обнаружения регрессий после изменений в движке или планировщике;
- валидации результатов относительно эталонных значений;
- сравнения производительности движков на реальных аналитических запросах.

TPC-H и TPC-DS широко признаны для этих целей. BenchBase предоставляет
единый драйвер: загружает схемы, генерирует данные нужного scale factor,
исполняет запросы и собирает метрики. Также поддерживает запуск через JDBC
(DuckDB подходит) и расширяем для Arrow-native движков (Acero).

## Решение

- Использовать **TPC-H** как основной бенчмарк для scan-heavy и
  aggregation-heavy нагрузок. Его 22 запроса покрывают star-join схемы,
  групповые агрегации и последовательные сканирования.
- Использовать **TPC-DS** как дополнительный бенчмарк для сложных
  аналитических запросов. Его 99 (или 103) запросов включают вложенные
  подзапросы, оконные функции, скошенные распределения данных и
  многотабличные соединения, нагружающие планировщик и движок.
- Запускать оба набора через **BenchBase**, чтобы не писать собственные
  benchmark harnesses, генераторы данных и сборщики метрик.
- Хранить конфигурацию BenchBase вместе с проектом для повторяемых
  запусков с едиными параметрами (scale factor, число потоков, количество
  итераций, warm-up раунды).
- Валидировать результаты DuckDB по эталонным данным TPC-H/TPC-DS.
  Для Acero валидировать по известным результатам DuckDB на тех же запросах.
- Добавить BenchBase-бенчмарки в CI для автоматического обнаружения
  регрессий на каждый коммит.

BenchBase выбран, потому что:

- поддерживает TPC-H и TPC-DS из коробки;
- нативно работает с DuckDB через JDBC — одним из движков проекта;
- в одном инструменте решает генерацию данных, создание схем, выполнение
  запросов и экспорт метрик;
- активно поддерживается сообществом и используется в аналогичных
  проектах баз данных.

## Рассмотренные альтернативы

### Вариант A: Собственные JMH микробенчмарки

Написать отдельные JMH бенчмарки для каждого типа запросов и каждого движка.

- **Плюсы**: Тонкий контроль над измерениями. Нет внешних зависимостей.
  Просто запускать в разработке.
- **Минусы**: Нет стандартного набора запросов — нужно придумывать с нуля.
  Нет генератора данных. Нет валидации по эталонным результатам. Сложнее
  воспроизвести в разных окружениях. Не измеряет end-to-end задержку на
  реалистичных данных.

### Вариант B: TPC-H / TPC-DS через собственную обвязку

Написать свой harness, который генерирует TPC-данные через `dbgen`/`dsdgen`
и выполняет запросы через собственный query path проекта.

- **Плюсы**: Полный контроль над загрузкой запросов, подстановкой параметров
  и сбором метрик. Глубокая интеграция с проектом.
- **Минусы**: Значительные трудозатраты. Нужно повторять логику подстановки
  параметров, warm-up и агрегации метрик, которую BenchBase уже реализует.
  Больше поверхности для багов в самом harness.

### Вариант C: BenchBase + TPC-H / TPC-DS (выбран)

Использовать BenchBase как готовый runner для обоих TPC-наборов.

- **Плюсы**: Проверенный инструмент со встроенной генерацией данных,
  подстановкой параметров и сбором метрик. DuckDB JDBC поддерживается
  напрямую. Расширяем для Acero через кастомный engine adapter, если
  потребуется. Активное сообщество.
- **Минусы**: BenchBase — Java-проект, добавляет зависимость и требует
  настройки под конкретное развёртывание. Acero требует кастомного адаптера
  (не JDBC движок).

## Последствия

- **Положительное**: Стандартизированные наборы запросов делают результаты
  сравнимыми с другими системами и между версиями проекта.
- **Положительное**: BenchBase устраняет необходимость писать и
  поддерживать собственный benchmark harness, генератор данных и сборщик
  метрик.
- **Положительное**: TPC-DS покрывает типы запросов (вложенные подзапросы,
  оконные функции, скошенные соединения), которых нет в TPC-H, выявляя
  слабые места движков, которые пропустили бы простые scan-бенчмарки.
- **Положительное**: JDBC-поддержка BenchBase даёт немедленное покрытие
  DuckDB-бенчмарков. Интеграция с CI реализуется стандартно.
- **Отрицательное**: TPC-DS генерирует большую схему (24 таблицы) и сложные
  планы запросов, что может замедлить начальную настройку и цикл итераций.
- **Отрицательное**: BenchBase добавляет Java-зависимость в инструментарий
  бенчмарков. Нужно отслеживать обновления версий.
- **Отрицательное**: Acero — не JDBC движок, нужен кастомный BenchBase
  adapter для его бенчмаркинга в том же фреймворке. Пока адаптер не написан,
  бенчмарки Acero остаются на отдельных JMH тестах или ручных запусках.
- **Будущее**: Если adapter model BenchBase окажется ограничивающей, проект
  может перейти на собственный harness, сохранив наборы запросов TPC-H/TPC-DS —
  инвестиции в привязку параметров и генерацию данных остаются.
