# ADR 0002: Распределение выполнения файлов между узлами

Status: Accepted
Date: 2026-07-07

## Решение

- Timestamp: 2026-07-07T10:30:00+07:00
- User: s.samokhin
- Вопрос: Как Flight SQL кластер должен распределять выполнение Parquet файлов
  между серверными узлами?
- Решение: Первой реализацией использовать планировщик, который учитывает
  размер файлов. Каждый узел получает набор файлов так, чтобы суммарный
  закэшированный размер файлов был примерно сбалансирован. Metadata с размером
  файлов кэшируется, чтобы планирование не читало filesystem metadata заново на
  каждый запрос.
- Обоснование: Количество файлов плохо описывает объём работы. Один большой
  Parquet файл может занять больше времени, чем много маленьких. Распределение
  по размеру простое, детерминированное и справедливее, чем round-robin по
  количеству файлов. Динамическое распределение по загруженности узлов
  планируется как следующий шаг.

## Контекст

Сервер может работать как несколько Arrow Flight SQL узлов над общим Parquet
dataset. Запрос может читать много Parquet файлов, и до начала выполнения эти
файлы нужно назначить доступным узлам.

У планировщика есть две ближайшие цели:

- достаточно ровно распределять работу между узлами;
- не делать дорогие чтения filesystem metadata на каждый запрос.

Сейчас самый стабильный сигнал для оценки работы — размер Parquet файла. Его
легко использовать до начала выполнения запроса, и он доступен для всех файлов.
Но это не идеальная оценка: размер файла не учитывает row-group pruning,
селективность фильтров, CPU pressure, memory pressure и текущую загруженность
узлов.

## Решение

- Для каждой таблицы собрать Parquet файлы, которые участвуют в запросе.
- Кэшировать metadata файлов, особенно размер, чтобы повторное планирование не
  обращалось к filesystem для тех же путей.
- Назначать файлы узлам через size-aware balancing:
  1. отсортировать файлы по размеру по убыванию;
  2. каждый следующий самый большой файл класть на узел с минимальным текущим
     суммарным размером назначенных файлов;
  3. выполнять на каждом узле его subset файлов локально, где это возможно.
- Назначение должно быть детерминированным для одного и того же набора файлов и
  узлов.
- File-size cache должен уметь обновляться или инвалидироваться, когда
  содержимое таблицы меняется.

Это эвристика планирования, а не финальная distributed execution модель.

## Дальнейшее развитие

Следующая версия планировщика должна учитывать динамические runtime-сигналы:

- текущую загруженность узла;
- количество активных запросов на узле;
- уже назначенные или сейчас сканируемые bytes;
- наблюдаемую скорость чтения;
- memory pressure;
- data locality, если filesystem может её предоставить.

Долгосрочная цель — гибридный planner: использовать закэшированный размер файла
как базовую оценку, а затем корректировать placement по live node load и
последним метрикам выполнения.

## Рассмотренные альтернативы

### Вариант A: Round-robin по количеству файлов

Назначать файлы узлам по одному, не учитывая размер.

- **Плюсы**: Очень просто и детерминированно.
- **Минусы**: Плохой баланс, когда размеры файлов отличаются. Один узел может
  получить несколько очень больших файлов, а другой — много маленьких.

### Вариант B: Статическое распределение по размеру (выбран)

Использовать закэшированные размеры файлов и жадно балансировать суммарные
bytes, назначенные каждому узлу.

- **Плюсы**: Просто, детерминированно, дёшево считать и намного честнее, чем
  file-count распределение, когда размеры файлов разные.
- **Минусы**: Всё ещё статическая эвристика. Она не знает, что один узел уже
  занят или что фильтрованный запрос может пропустить большую часть row-group в
  крупном файле.

### Вариант C: Динамическое распределение по загруженности

Назначать работу по live metrics: CPU load, memory pressure, активные запросы и
текущая скорость сканирования.

- **Плюсы**: Потенциально лучшая производительность при concurrent workload и
  skew.
- **Минусы**: Больше moving parts. Нужны надёжные метрики узлов, периодическое
  обновление, обработка отказов и аккуратное тестирование, чтобы планировщик не
  принимал нестабильные решения.

## Последствия

- **Положительное**: Планирование запросов становится справедливее, чем
  распределение по количеству файлов.
- **Положительное**: Кэшированная metadata файлов уменьшает overhead
  планирования для повторных запросов.
- **Положительное**: Дизайн оставляет место для динамического load-aware
  scheduling в будущем.
- **Отрицательное**: Размер файла — только оценка, а не точная стоимость работы.
  Фильтры, row-group pruning, compression и column projection могут сильно
  изменить реальное время выполнения.
- **Отрицательное**: Metadata cache требует стратегии refresh/invalidation,
  когда файлы добавляются, удаляются или перезаписываются.
- **Будущее**: Добавить node-load metrics и использовать их вместе с
  закэшированным размером файлов, чтобы assignment стал adaptive, а не чисто
  статическим.
