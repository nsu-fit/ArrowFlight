# ADR 0003: Внедрение зависимостей через Dagger

Status: Accepted
Date: 2026-07-13

## Решение

- Timestamp: 2026-07-13T12:00:00+07:00
- User: vladimir
- Вопрос: Какой подход к внедрению зависимостей использовать в Flight SQL
  сервере, учитывая необходимость тестируемости, явного wiring'а и
  минимального runtime-оверхэда для Arrow-native памяти?
- Решение: Использовать Dagger для compile-time dependency injection.
- Обоснование: Dagger обеспечивает compile-time безопасность, лёгкий runtime
  (~0.5 MB), быстрый старт (~5 ms) и отсутствие рефлексии — критично для
  слоя нативной памяти Arrow. Spring Boot был бы overkill (50 MB, classpath
  scanning, runtime proxies). Guice использует рефлексию в runtime, добавляя
  оверхэд при старте и потенциальные проблемы с off-heap памятью Arrow.

## Контекст

Изначально в проекте не было DI-фреймворка. Все зависимости создавались через
`new` внутри `HadoopArrowFlightServer.start()` и дублировались в тестах.

Это вызывало ряд проблем:

- **Невозможность unit-тестирования** — классы вроде `ParquetManager`
  (позже разделён на `ExecutionService`, `MetadataService`, `ParquetAdapter`)
  были жёстко закодированы в точке входа. Тесты не могли замокать отдельные
  компоненты.
- **Неявный граф зависимостей** — чтобы понять, кто от кого зависит, нужно
  было читать весь `start()` и высматривать `new`.
- **Дублирование wiring'а** — каждый тест копипастил 30–50 строк ручного
  создания (`ParquetManager`, `HazelcastInstance`, `RootAllocator`,
  `FlightSqlProducer` и т.д.).
- **Нет гарантий синглтонов** — ничто не предотвращало случайное создание
  двух экземпляров `BufferAllocator`.

Во время рефакторинга `restructure2` был внедрён Dagger для решения этих
проблем.

## Сравнение: Dagger vs ручной DI

| Критерий | Ручной (main) | Dagger |
| :--- | :--- | :--- |
| Compile-time безопасность | ❌ (null по невнимательности) | ✅ |
| Стартовый оверхэд | 0 ms | ~5 ms (генерация кода) |
| Размер библиотеки | 0 MB | ~0.5 MB |
| Тестируемость | ❌ (жёсткий wiring) | ⚠️ (ручной `TestFlightServerHelper`) |
| Гарантии синглтонов | ❌ | ✅ (`@Singleton`) |
| Явный граф зависимостей | ❌ | ✅ (`ServerModule`) |

**Оценка: 9/10** — значительное улучшение относительно отсутствия DI. Wiring
явный в `ServerModule`, синглтоны гарантированы через `@Singleton`, граф
зависимостей виден с первого взгляда.

## Сравнение: Dagger vs Spring Boot

| Критерий | Spring Boot | Dagger |
| :--- | :--- | :--- |
| Compile-time безопасность | ❌ (только runtime) | ✅ |
| Стартовый оверхэд | ~10 s (classpath scan) | ~5 ms |
| Размер библиотеки | ~50 MB | ~0.5 MB |
| AOP / метрики | ✅ (Actuator, Micrometer) | ❌ |
| Кривая обучения | Высокая | Средняя |
| Подходит для проекта | ⚠️ (overkill) | ✅ |

**Оценка: 7.5/10** — Spring Boot был бы overkill. В проекте нет web-слоя,
транзакций, JPA или scheduling. Dagger покрывает 100% потребностей DI с 1%
веса.

## Сравнение: Dagger vs Guice

| Критерий | Guice | Dagger |
| :--- | :--- | :--- |
| Compile-time безопасность | ❌ (runtime reflection) | ✅ |
| Стартовый оверхэд | ~500 ms | ~5 ms |
| Размер библиотеки | ~2 MB | ~0.5 MB |
| AOP | ⚠️ (AOP Alliance) | ❌ |
| Тестируемость | ✅ (`Modules.override`) | ⚠️ (ручной) |
| Подходит для проекта | ⚠️ (runtime overhead) | ✅ |

**Оценка: 7.5/10** — Guice — runtime-фреймворк, использующий рефлексию и
динамические прокси. Для высокопроизводительного Arrow-сервера с native
памятью рефлексия нежелательна. Dagger генерирует код на этапе компиляции
с нулевым runtime-оверхэдом.

## Решение

- Использовать **Dagger** как DI-фреймворк.
- Все синглтоны wiring'ятся в `ServerModule` через `@Provides` и
  `@Singleton`.
- Сгенерированный `DaggerServerComponent` собирается в
  `HadoopArrowFlightServer.start()`.
- В тестах используется `TestFlightServerHelper` с ручным wiring (Dagger в
  тестах не применяется).

Dagger — оптимальный компромисс: compile-time безопасность без раздувания
Spring Boot и runtime-оверхэда Guice.

## Последствия

- **Положительное**: Граф зависимостей явный в `ServerModule`. Больше не
  нужно искать `new` по всему коду.
- **Положительное**: `@Singleton` гарантирует единственный экземпляр
  `BufferAllocator`, `HazelcastAdapter`, `FileSystem` и других тяжёлых
  объектов.
- **Положительное**: Ошибки в графе зависимостей (отсутствующие провайдеры,
  циклические зависимости) обнаруживаются на этапе компиляции.
- **Положительное**: Лёгкий runtime (~0.5 MB) и быстрый старт (~5 ms).
- **Отрицательное**: Тесты всё ещё требуют ручного wiring в
  `TestFlightServerHelper`. Нет `@MockBean` или `Modules.override`.
- **Отрицательное**: Нет AOP, lifecycle-колбэков, метрик. Пока они не нужны.
- **Будущее**: Если проект обрастёт HTTP-endpoints, health checks, scheduled
  tasks или метриками — Spring Boot станет жизнеспособной альтернативой. Пока
  этого нет, Dagger — правильный выбор.
